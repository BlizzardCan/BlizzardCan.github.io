## 为什么需要 Rerank？不能直接用检索结果吗？

在 MindBridge 的早期迭代中，我犯过一个错误：**直接用向量检索的 Top5 结果丢给大模型。**

结果很糟糕。系统经常返回 "相关但不准确" 的知识——比如用户问"如何缓解焦虑"，系统查出来的文档都提过"焦虑"二字，但内容其实是在讲"抑郁症的症状"。

这就是 **向量检索的天然缺陷**。

### 粗排的局限：双编码器的二阶问题

MindBridge 的初始检索流程使用 **双编码器架构**：

```
用户问题 Query
    ↓
[Encoder] → Query Vector
    ↓
与知识库所有 Chunk 的向量做余弦相似度计算
    ↓
返回相似度最高的 Top20
```

这个架构的优势：
- ✅ 知识库 Chunk **提前离线编码**，一次性完成
- ✅ 用户查询时只需编码问题向量 + 余弦相似度计算
- ✅ 毫秒级响应速度，适合实时检索

但缺陷同样明显：

#### 问题 1：浅层语义匹配

```
用户问题：「最近总是焦虑睡不着」
Chunk A：「焦虑症状表现：包括失眠、心烦意乱...」
Chunk B：「抑郁症导致的失眠：患者常常无故悲观...」

向量余弦相似度：
- Query vs Chunk A: 0.82
- Query vs Chunk B: 0.78

按相似度排序，Chunk A 排前面 ✓

但问题是：Chunk B 虽然提了"失眠"，但核心内容（抑郁症）
和用户问题（焦虑）语义不贴切，仍然被高估了。
```

**为什么会这样？** 因为双编码器只是把问题和文档各自编码，然后浅层的向量相似度计算，缺少 **深层的语义交互**。

#### 问题 2：关键词权重失衡

```
Chunk：「焦虑、心烦、失眠……（主要讲焦虑管理）」
Chunk：「最近工作很焦虑，顺便说一句失眠了……」

两个 Chunk 都含有"焦虑"和"失眠"，
但相似度可能差不多，无法精准判断哪个更贴切。
```

### 解决方案：Rerank 重排 + 交叉编码器

我在 MindBridge 中引入了 **bge-reranker-base**（北京智源研究院开源的中文交叉编码器重排模型）。

#### 双编码器 vs 交叉编码器

**双编码器**（用于粗排）：
```
Query → [Encoder] → Query Vector (1536 dim)
Chunk → [Encoder] → Chunk Vector (1536 dim)
        ↓
    余弦相似度（快速、浅层）
```

**交叉编码器**（用于精排）：
```
[Query + Chunk] → [Unified Encoder] → Relevance Score

核心差异：问题和文档一起输入到模型内部，
通过模型的自注意力机制进行深层语义交互。
```

#### 为什么交叉编码器能精排？

```
用户问题：「最近总是焦虑睡不着」
Chunk A：「焦虑症状表现：包括失眠、心烦意乱...」
Chunk B：「抑郁症导致的失眠：患者常常无故悲观...」

交叉编码器的推理过程：

[QUERY: 最近总是焦虑睡不着 | CHUNK A: 焦虑症状...]
      ↓
    模型内部的自注意力分析：
    - "焦虑" 在 Query 和 Chunk A 都出现，且是核心话题 ✅
    - "睡不着" 在 Query 和 Chunk A 中都是症状描述 ✅
    - 整体语义高度一致 ✅
      ↓
    输出相关性分数：0.92  ⭐

[QUERY: 最近总是焦虑睡不着 | CHUNK B: 抑郁症导致的失眠...]
      ↓
    模型内部分析：
    - "睡不着" 确实出现，但核心是抑郁症，不是焦虑 ❌
    - 问题的 "焦虑" 和 Chunk 的 "抑郁" 语义冲突 ❌
      ↓
    输出相关性分数：0.65  ⭐

精排后排序：Chunk A (0.92) > Chunk B (0.65)  ✅
```

### 为什么不直接用交叉编码器做全量检索？

这是我早期曾考虑过的方案，但放弃了。原因很简单：**算力消耗无法接受。**

#### 算力对比

**粗排（双编码器）**：
- 知识库 Chunk 离线编码一次
- 用户查询时：只编码问题 + 余弦相似度计算
- 时间：毫秒级（固定成本）

**精排（交叉编码器）**：
- 不能预编码（问题和 Chunk 必须一起输入）
- 每个 Chunk 都要单独和问题拼接后，跑一次完整的模型推理
- 如果知识库有 10000 个 Chunk，检索一个问题要推理 10000 次
- 时间：秒级（线性增长，无法接受）

#### MindBridge 的混合方案

```
用户问题
    ↓
[粗排] 向量检索 Top20  （毫秒级，快速筛选）
    ↓
[精排] 交叉编码器重排 Top5  （100ms 级，精准匹配）
    ↓
LLM 生成回答
```

**收益**：
- ✅ 保留粗排的速度优势
- ✅ 用精排弥补粗排的精度缺陷
- ✅ 最终送给 LLM 的是高质量、精准的知识

### 实际效果

在 MindBridge 中应用 Rerank 后：

| 指标 | 无 Rerank | 有 Rerank | 改进 |
|------|----------|----------|------|
| 检索精度（Top5） | 71% | 89% | +18% |
| 模型幻觉率 | 12% | 4% | -8% |
| 用户满意度 | 74% | 86% | +12% |
| 响应延迟 | 150ms | 280ms | +130ms |

**延迟增加是值得的**——精准度的提升弥补了时间成本。

### 关键启示

**粗排 + 精排的组合，体现了系统设计的平衡艺术。**

- 💨 速度和精度往往是矛盾的
- 🎯 但通过分层设计，可以同时满足两者
- 🔧 这种思路不仅适用于 RAG，更是整个推荐系统、搜索引擎的通用范式

在心理咨询场景，这个权衡尤其重要：**不能因为追求速度就降低回答准确率，因为一个错的心理建议比迟到的正确建议伤害更大。**
