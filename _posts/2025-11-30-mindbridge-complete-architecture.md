## 从需求到产品：MindBridge 的完整演进

经过 11 个月的探索和优化，今天我想把 MindBridge 整个项目的架构设计、技术选型、问题应对都系统化地总结一遍。

这不仅是技术总结，更是对 **如何把 AI 能力真正落地到应用场景** 的思考。

---

## 第一步：为什么需要 MindBridge？

### 痛点分析

传统的心理咨询AI系统存在几个核心问题：

1. **规则引擎太脆弱**
   - 只能匹配固定的关键词和预定义的回复
   - 无法理解用户的真实诉求
   - 用户体验差

2. **大模型直接部署会幻觉**
   - 让模型直接回答心理问题，会编造不负责任的建议
   - 这在心理咨询场景是不能接受的

3. **缺乏专业性约束**
   - 模型的回答不够专业，不符合心理咨询伦理
   - 用户信任度低

**解决方案：用 RAG + LLM 构建可靠的心理咨询助手**

---

## 第二步：技术架构演进

### 四阶段迭代

#### 阶段 1：规则引擎 → Qwen 2.5-7B 大模型

```
规则引擎时代：
用户输入 → 关键词匹配 → 预定义回复

缺点：回答生硬，无法理解复杂表述
```

**升级**：引入 **Qwen 2.5-7B** 作为核心对话引擎

```
用户输入 → Qwen 2.5-7B → 自然流畅的回答

优点：对话更自然、更有同理心
```

#### 阶段 2：LLM 直接回答 → RAG 增强

```
问题：Qwen 直接回答会幻觉
"我今天很烦" 
→ Qwen 可能编造："建议你吃一顿大餐，这样心情会好"（不专业）
```

**升级**：接入 RAG，用心理咨询知识库约束模型

```
用户问题 → RAG 检索相关知识 → Qwen 基于知识回答
           ↓
         知识库（心理咨询文档）

效果：回答有依据、有底线、有专业性
```

#### 阶段 3：通用 Embedding → 风险识别微调

```
问题：系统只是被动回答问题，无法主动识别心理风险

用户说："最近睡眠很差，总是想不开"
→ 系统没有识别到这是抑郁症风险信号
```

**升级**：用 **LoRA 微调**优化心理风险识别能力

```
基础 Embedding 模型
       ↓
   LoRA 微调
       ↓
   专业心理风险识别模型
       
效果：系统可以主动识别"自伤"、"抑郁风险"等关键信号
```

#### 阶段 4：分散模块 → SpringAI 统一框架

```
问题：
- LLM 模块散乱
- RAG 模块散乱  
- 记忆管理散乱
- 工具调用散乱

无法形成完整系统
```

**升级**：用 **SpringAI** 框架统一集成

```
SpringAI
  ├─ LLM Integration（Qwen）
  ├─ RAG Module（向量库 + 检索）
  ├─ Memory Management（对话上下文）
  ├─ Tool Calling（如转接心理咨询师）
  └─ Risk Detection（LoRA 微调的识别器）

优点：
- 架构清晰
- 各模块解耦
- 易于扩展和维护
```

## 第三步：RAG 系统的完整设计

我把 RAG 系统分为 **离线阶段** 和 **在线阶段**。

### 离线阶段：一次性构建知识库

```
1. 数据准备
   心理咨询专业文档（PDF、Word）
        ↓

2. 语义切块
   把长文档按语义边界切成 500-800 字的 Chunk
   （而不是简单的按字数切）
        ↓

3. Embedding 向量化
   用 text-embedding-3-small 模型
   把每个 Chunk 转换成 1536 维向量
   （这一步离线完成，效率最高）
        ↓

4. 向量存储
   把向量和原文本一起存入 Chroma 向量库
   （可以持久化到磁盘，也可以用云向量库）

结果：知识库就绪，可以快速检索
```

### 在线阶段：处理每个用户提问

```
用户提问：「最近特别焦虑睡不着」
        ↓

1. 意图判断
   这是心理咨询问题吗？
   - 是 → 触发 RAG
   - 否（是闲聊） → 直接让 Qwen 回答
   - 模糊 → 反问用户澄清
        ↓

2. 查询预处理
   去掉虚词：「啊」「吧」「超级」
   情绪词归一化：「崩了」→「情绪失控」
   场景词归一化：「学不进去」→「学业压力」
        ↓

3. Query 向量化
   把预处理后的问题用同样的 text-embedding-3-small 向量化
        ↓

4. 粗排检索
   Chroma 向量库做余弦相似度计算
   返回最相似的 Top20 Chunk
        ↓

5. 精排重排
   用 bge-reranker-base 交叉编码器
   对 Top20 做二次语义匹配
   筛选出最贴合的 Top5 Chunk
        ↓

6. Prompt 拼接
   系统指令 + 对话上下文（最近10轮）
   + 用户当前问题 + Top5 知识库内容
        ↓

7. LLM 生成回答
   Qwen 2.5-7B 接收 Prompt
   基于知识库内容生成回答
        ↓

8. 风险识别
   LoRA 微调的识别器同步检测
   如发现"自伤"等高风险信号
   → 立即转接人工客服
        ↓

回答返回给用户
```

### 监控与迭代

```
线上运行中：
- 跟踪检索精准度
- 监控幻觉率
- 收集用户反馈
- 定期复盘失败案例
        ↓

每周迭代：
- 补充新的知识库内容
- 调整混合检索权重
- 优化 Rerank 逻辑
- 更新 LoRA 微调模型
```

## 第四步：核心模块详解

### 1. Qwen 2.5-7B：对话引擎

**为什么选 Qwen 而不是其他模型？**

- ✅ 中文能力强（心理咨询是中文场景）
- ✅ 7B 参数量平衡了性能和成本
- ✅ 可以本地部署（MindBridge 是校园项目，隐私要求高）
- ✅ 开源、有社区支持

### 2. text-embedding-3-small：向量化模型

**为什么选这个模型？**

- ✅ 1536 维度，足够表达心理语义
- ✅ 中文表现好
- ✅ 业界认可度高
- ✅ 成本合理

### 3. Chroma 向量库：知识存储

**为什么选 Chroma？**

- ✅ 轻量级，无需独立服务
- ✅ 支持本地持久化
- ✅ 原生支持余弦相似度
- ✅ Python 集成简单

### 4. bge-reranker-base：精排模型

**为什么要单独的重排模型？**

- ✅ 中文交叉编码器，专业
- ✅ 开源且轻量
- ✅ 精度明显优于向量相似度
- ✅ 能解决粗排的精度问题

### 5. LoRA 微调：风险识别

```
基础模型无法识别心理风险信号
        ↓
用标注数据（自伤、抑郁、焦虑等）微调
        ↓
LoRA 方式（只调整少量参数，成本低）
        ↓
识别精度从 60% → 90%
```

## 第六步：关键挑战与解决方案

### 挑战 1：LLM 幻觉

**问题**：模型编造不存在的心理建议

**解决方案**：
- 严格的系统 Prompt 约束
- RAG 提供知识库支撑
- 定期监控幻觉案例

**结果**：幻觉率从 25% → 4%

### 挑战 2：线上线下不一致

**问题**：离线测试 90%，上线后 65%

**解决方案**：
- 用真实数据反哺测试集
- 优化查询预处理
- 迭代意图识别规则

**结果**：线上表现从 65% → 81%

### 挑战 3：性能和精度的平衡

**问题**：精排模型精度高但速度慢

**解决方案**：
- 粗排 + 精排的混合方案
- 粗排快速筛选，精排精准匹配

**结果**：
- 精度：78% → 89%
- 延迟：150ms → 280ms（可接受）

## 总结：MindBridge 的核心竞争力

### 技术层

- ✅ 完整的 RAG 架构
- ✅ 专业的心理风险识别
- ✅ 可控的、可信的回答生成

### 业务层

- ✅ 适配校园心理服务场景
- ✅ 缓解心理咨询师的工作压力
- ✅ 为学生提供 24/7 的初级咨询服务

### 个人学习层

MindBridge 让我学到了：

1. **技术不是目的，场景才是**
   - RAG、LLM 都是工具
   - 真正的价值在于理解场景、解决真实问题

2. **系统工程的重要性**
   - 不是单个模型的精度高就行
   - 整个流程的每一个环节都会影响最终效果

3. **持续迭代的必要性**
   - 第一版永远不完美
   - 线上反馈是最好的老师

4. **细节决定成败**
   - Prompt 怎么写
   - Chunk 怎么切
   - TopK 怎么选
   - 这些看似小的决定，累积起来决定了系统的质量
