## 离线测试都过了，为什么上线还是翻车？

这是我在 MindBridge 部署后遇到的最扎心的问题。

离线测试集里，系统评估指标都达到 90%+ 的准确率。但上线 3 天后，收到大量用户反馈：**"系统回答牛头不对马嘴"、"感觉不懂我的问题"。**

这就是著名的 **线上线下不一致** 问题。

这篇文章记录我怎么把“测试集很好、线上翻车”的问题拆开看，并用真实日志和 query 预处理去对齐数据分布。

<!--more-->

### 根本原因：3 个数据分布差异

#### 1. 用户提问的口语化程度远超测试集

**离线测试集**（规整、标准化）：
- "我最近因为学业压力大，导致睡眠不足，如何缓解？"

**真实线上提问**（碎片化、带情绪）：
- "我好烦啊学不进去，昨天又没睡好，怎么办啊"
- "最近总睡不着心里堵得慌"
- "emo、崩了、超级累"

离线句式工整，上线提问口语化、碎片化、充满情绪词汇——**Embedding 向量检索直接跑偏。**

#### 2. 检索噪声：大量无关词汇干扰

用户提问中包含大量虚词、语气词：

"啊、吧、超级、真的、嗯、呃、可是…"

这些词汇虽然不影响人类理解，但会 **稀释核心语义**，让向量表示噪声满满。

比如一个包含 30% 虚词的提问，Embedding 生成的向量可能完全反映不了用户的真实心理诉求。

#### 3. 用户意图模糊：系统难以判断

线上很多问题的 **意图边界不清**：

- 用户只说"我心里不舒服" → 到底是专业心理问题还是普通情绪吐槽？
- 系统可能误判 → 要么不触发 RAG（漏知识支撑）、要么过度解读（错误检索）

结果：线上表现远不如离线。

### 解决方案：4 个方向

#### 1. 用线上真实数据反哺离线测试集

**关键操作**：定期从 MySQL 导出线上失败案例

```
SELECT * FROM dialogue_records 
WHERE result = 'failed' OR user_satisfaction = 'low'
LIMIT 100;
```

把这些 badcase 人工标注后，**加入离线测试集**，让数据分布对齐。

好处：
- 离线评估数据和线上完全一致
- 后续迭代有真实案例可对标

#### 2. 加一层用户提问预处理

在检索前，对 query 做 **清洗 + 归一化**：

```
用户输入：我emo了崩了，最近超级烦学不进去啊

清洗步骤：
1. 过滤虚词：去掉 "啊、超级、了"
2. 情绪词归一化：emo/崩了 → 情绪失控
3. 场景词归一化：学不进去 → 学业压力

处理后：情绪失控 学业压力
```

这样处理后的 query 向量和知识库的匹配度大幅提升。

#### 3. 优化意图识别：细化路由规则

不能简单的"是/不是心理问题"二分，要 **细化分类**：

- **心理专业问题**（触发 RAG）："如何缓解焦虑症状？"
- **模糊问题**（反问澄清）："我心里不舒服" → 系统反问"能具体说说吗？"
- **闲聊**（直接回答）："今天天气怎样？"

特别是模糊问题，不要盲目路由，要 **先反问用户澄清**，再判断是否触发 RAG。

#### 4. 定期复盘线上失败案例

每周梳理：
- 漏召的问题（系统该查到却没查到）
- 答非所问的问题（检索了错误的知识）
- 重复提问的问题（系统上一次回答不满意）

针对性调整：
- 混合检索的权重配置
- 知识库的切块方式
- Rerank 重排逻辑

### 实际效果

在 MindBridge 中应用这套方案后：

| 指标 | 优化前 | 优化后 | 提升 |
|------|-------|-------|------|
| 用户满意度 | 62% | 81% | +19% |
| 系统回答准确率 | 71% | 88% | +17% |
| 漏召问题占比 | 28% | 9% | -19% |

## 关键启示

**线上线下不一致，本质上是数据分布的差异。** 

单纯优化离线指标是不够的，真正的考验是 **如何快速发现线上问题，并用真实数据持续迭代**。

心理咨询场景更是如此——真实用户的语言、情绪、诉求方式，远比任何标注数据集更复杂、更多变。

我学到的是：**打磨一个好的 RAG 系统，就像做心理咨询一样，需要不断倾听、不断调整，永远不能自满。**
